# ガイドライン

## 新しい検証の準備を始めるときに守ること

- <検証の概要を短くまとめた英語> のフォルダを新規作成すること
- <検証の概要を短くまとめた英語> のフォルダ配下に、実際に検証を行うエージェント向けのルールを記載するための<検証の概要を短くまとめた英語>.AGENTS.md を作成すること
- <検証の概要を短くまとめた英語>.AGENTS.md の冒頭に、本 AGENTS.md の「エージェントが守るべきこと」セクションをコピペして記載すること

## エージェントが守るべきこと

### 利用ツール

- Azure を CLI で操作する際には Azure PowerShell を利用する。Azure CLI の利用は禁止
- `Invoke-AzRestMethod` および `Invoke-AzRest` は可能な限り利用しない。Azure の操作には Azure PowerShell の各モジュールに含まれるコマンドを利用すること
- `Invoke-AzVMRunCommand` などの「実行コマンド（Run Command）」機能を、WinRM を有効化するため以外で使用してはならない
- Windows VM へのリモートコマンド実行には Remote PowerShell（WinRM HTTPS） を使う
  - セッション確立: `New-PSSession -ComputerName <IP> -Credential <cred> -UseSSL -SessionOption (New-PSSessionOption -SkipCACheck -SkipCNCheck -SkipRevocationCheck)`
  - コマンド実行: `Invoke-Command -Session <session> -ScriptBlock { ... }`

### 検証時の注意点

- GUI による操作は禁止。CLI で検証を完遂すること
- Azure PowerShell による操作を始める前に、ログインしているコンテキスト（サブスクリプション・テナント）が正しいかどうかをユーザに確認すること
- 各クラウド操作コマンドの実行後に、必ず状態確認コマンドを挟む（作成・更新・割り当て・削除すべて対象）
- ブログの記事に実行結果を転記しやすいように、Azure PowerShell による Azure の操作をパイプ連結、長いワンライナー、実行用スクリプトによる一括実行を禁止する
- 実行したコマンドとその出力を ./outputs フォルダ配下にログファイルに記録する。ログファイルは「ブログに証跡として転記するため」と「人間がエージェントによる処理の過程とその結果を後から確認できるようにするため」に利用される
- 検証結果を記載したマークダウンファイルを ./outputs フォルダ配下内に保存する。ファイル名は <検証内容を表す英語>-report.md とする
- 手順を実行して発生したエラーに対処するために手順を変更した場合には AGENTS.md 内の手順を修正すること

### Azure リソースの設定

- ストレージアカウントへのアクセスに SAS トークンを使用することを禁止する。VM からのパッケージ取得には必ずシステム割り当てマネージド ID（Storage Blob Data Reader ロール）を使うこと
- ストレージアカウントのアクセスキー認証を無効化しなければならない。つまりストレージアカウントの認証では Entra ID 認証を使うこと
- ストレージアカウントのパブリックアクセスを無効化しなければならない。つまりストレージアカウントへのアクセスにはプライベートエンドポイントを利用すること
- ストレージアカウントのパブリックアクセスを一時的であっても有効化することを禁止する。切り分けや動作確認を目的とした一時的な有効化も含めて禁止
- VM 作成時は `New-AzVM` を呼ぶ直前に必ず `Set-AzVMBootDiagnostic -VM $vmConfig -Enable`（`-StorageAccountName` なし）を実行すること。省略するとランダム名のストレージアカウントがサブスクリプションに自動作成されるため禁止する

### ブログを作成する際の注意点

- ブログ記事を記載する際には「ですます調」を使用すること。
- 証跡としてコマンドの出力を記載する際には、出力だけでなく実行したコマンドも含めること
- ブログ出力時に `**〜**` による太字（bold）を使用してはならない
- セクションとセクションの間を `---` で区切らないこと
- 記事の末尾（notice ブロックの直前）に参考リンクのセクションを追加すること
- ブログ記事の冒頭には必ず以下の YAML front matter を付与すること
- 長いコマンドをブログに記載する際には、実際のコマンドを、PowerShell でサポートされる ` を利用した改行を用いて見やすくする
- 以下を記載する際には ID やリソース名の部分を匿名化する
  - サブスクリプション ID
  - リソースの FQDN

```yaml
---
title: <記事タイトル>
author: kongou_ae
date: <YYYY-MM-DD>
url: /archives/<YYYY>/<MM>/<slug>
categories:
  - azure
---
```

- 記事の一番最後に必ず以下の notice ブロックを挿入すること

```
{{%/* notice note */%}}

- 当サイトは個人のブログです。このブログに示されている見解や意見は個人的なものであり、所属組織の見解や意見を表明するものではありません。
- 公開情報を踏まえて正確な情報を掲載するよう努めますが、その内容の完全性や正確性、有用性、安全性、最新性について一切保証しません。
- 添付文章やリンク先などを含む本サイトの内容は作成時点でのものであり、予告なく変更される場合があります。

{{%/* /notice */%}}
```
